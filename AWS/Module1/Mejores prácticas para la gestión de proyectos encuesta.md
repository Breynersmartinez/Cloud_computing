Piense en un proyecto en el que haya trabajado con éxito. ¿Qué buenas prácticas siguió en este proyecto? En su respuesta, conteste a las siguientes preguntas:

¿Decidió que el proyecto era viable antes de empezarlo?  En caso afirmativo, ¿cómo determinó la viabilidad del proyecto?

¿Cuáles fueron las justificaciones del proyecto? ¿Qué beneficios ofreció el proyecto finalizado?

¿Empezó el proyecto con una buena idea de los recursos que requeriría? ¿Cuáles fueron los diferentes tipos de recursos que necesitó planificar, como horas de trabajo, materiales y otros?

¿Tenía el proyecto unos objetivos bien definidos antes de empezar?  ¿Cumplió los objetivos del proyecto? ¿Necesitó cambiar los objetivos durante el proyecto?

¿Entregó el proyecto en varios hitos? ¿O entregó el proyecto con un único gran hito?

Después de completar el proyecto, ¿utilizó alguna de las lecciones que aprendió en futuros proyectos?

A modo de comparación, también puede responder a estas mismas preguntas sobre un proyecto fracasado.

Repuestas:

Decidió que el proyecto era viable antes de empezarlo?
En caso afirmativo, ¿cómo determinó la viabilidad del proyecto?
Sí, antes de iniciar el proyecto realicé un análisis de viabilidad técnica, económica y de tiempo.
Viabilidad técnica: evalué si el equipo tenía los conocimientos necesarios en el stack tecnológico (framework, base de datos, arquitectura, etc.).
Viabilidad económica: estimé el costo en horas de desarrollo frente al beneficio esperado.
Viabilidad de tiempo: definí un cronograma preliminar con estimaciones por módulos.
Viabilidad operativa: validé que el sistema resolviera una necesidad real del usuario.

Con base en estos factores, se concluyó que el proyecto era viable.
¿Cuáles fueron las justificaciones del proyecto?
¿Qué beneficios ofreció el proyecto finalizado?
El proyecto se justificó porque buscaba:
Automatizar procesos manuales.
Reducir errores humanos.
Centralizar la información.
Mejorar la experiencia del usuario.

Los beneficios obtenidos fueron:
Mayor eficiencia operativa.
Reducción de tiempos en procesos clave.
Mejor control y trazabilidad de la información.
Escalabilidad para futuras mejoras.

¿Empezó el proyecto con una buena idea de los recursos que requeriría?
¿Qué tipos de recursos necesitó planificar?
Sí, desde el inicio se estimaron los siguientes recursos:
Horas de desarrollo (backend, frontend, pruebas).
Recursos humanos (desarrolladores, tester si aplicaba).
Infraestructura tecnológica (servidor, base de datos, hosting).
Herramientas de desarrollo (control de versiones, gestor de tareas).
Tiempo para pruebas y despliegue.

Durante el desarrollo se ajustaron algunas estimaciones, pero la planificación inicial permitió mantener el proyecto bajo control.
 ¿Tenía objetivos bien definidos antes de empezar?
¿Se cumplieron? ¿Cambió alguno?
Sí, el proyecto inició con objetivos claros y medibles, como:
Desarrollar funcionalidades específicas.
Cumplir con ciertos requisitos técnicos.
Entregar dentro del tiempo estimado.

La mayoría de los objetivos se cumplieron. Algunos fueron ajustados debido a nuevos requerimientos del cliente o mejoras detectadas durante el desarrollo. Estos cambios se gestionaron mediante control de alcance.
¿Entregó el proyecto en varios hitos o en un único gran hito?**
El proyecto se desarrolló por hitos o iteraciones, permitiendo:
Entregar módulos funcionales progresivamente.
Recibir retroalimentación temprana.
Reducir riesgos técnicos.

Este enfoque permitió realizar ajustes antes de la entrega final.
Después de completar el proyecto, ¿utilizó alguna lección aprendida en futuros proyectos?**
Sí, algunas lecciones aprendidas fueron:
Mejorar la estimación de tiempos para tareas complejas.
Definir mejor los requisitos antes de iniciar el desarrollo.
Incluir pruebas desde etapas tempranas.
Mantener comunicación constante con el cliente o usuario final.

Estas lecciones ayudaron a optimizar la planificación y ejecución en proyectos posteriores.
